# 产品需求文档 (PRD)

## 1. 产品背景与目标
用户需要一个运行在本地的“大模型协议转换网关”。该软件需具备原生操作系统的图形界面 (Native UI)，且运行时不需要依赖网络翻墙代理。
其核心功能是将外部客户端（如 Codex/Cursor，以及 Claude Desktop）发出的标准模型请求，无缝转换为 Nvidia NIM API 的请求格式，并处理返回结果（包括流式输出）。

## 2. 技术架构选型
- **底层架构**：采用 **Go 语言**。Go 的高并发、低内存消耗特性非常适合作为网关处理高频的流式文本请求，且打包后为纯二进制文件。
- **UI 框架选型**：采用 **Wails** 框架。Wails 允许使用 Go 编写后端，使用前端标准 Web 技术 (HTML/CSS/JS) 构建极其轻量、跨平台的原生桌面客户端。这使我们能够完美落地要求的 "Apple Minimal 2026" 审美规范（原生毛玻璃、设计令牌、弹性阻尼动效）。

## 3. 核心功能特性

### 3.1 图形用户界面 (UI)
- **配置面板**：允许用户填写并保存 `Nvidia API Key`、绑定的本地端口（默认 `8080`）。
- **运行状态**：一键启动/停止网关服务，显示当前的运行状态（红绿指示灯）。
- **日志监控面板**：实时滚动显示拦截到的请求记录（例如：来源客户端、处理状态、消耗时间）。
- **视觉要求**：严格执行 Apple UI 2026 标准，双模兼容，界面无多余线条，高内聚低耦合。

### 3.2 核心业务逻辑 (Go 后端)
- **OpenAI 通道 (`/v1/chat/completions`)**：接收 Codex 等客户端请求，提取参数，向 `https://integrate.api.nvidia.com/v1/chat/completions` 发起请求并支持 Stream 数据流式透传。
- **Anthropic 协议转换通道 (`/v1/messages`)**：
  - 接收 Claude 客户端请求，将 `System`, `Messages` 数组提取并映射为 OpenAI 格式。
  - 向 Nvidia API 请求。
  - 接收 Nvidia 的 Stream 数据块，将其逆向封装为 Anthropic 约定的 `content_block_start`, `content_block_delta`, `message_stop` 等事件，通过 SSE 响应给 Claude 客户端。

## 4. 交付约束
- “谁污染谁治理”，代码极致精简。
- 前后端逻辑严格分离：UI 只负责展示和配置，Go 侧专注协议转发和流处理。
- 不需要引入重量级的第三方框架，能原生的尽量原生。
